### **Функциональные требования**

|**№**|**Действующие лица или системы**|**Use Case**|**Описание**|
| :-: | :- | :- | :- |
|1|Сотрудник склада|Загрузка отчёта об остатках|Пользователь формирует CSV-файл на основе Excel-шаблона и загружает его через веб-интерфейс ERP. Файл проходит валидацию и сохраняется в GCS.|
|2|Spring Batch Job|Чтение данных из CSV|FlatFileItemReader построчно читает файл product-data.csv из файлового хранилища и передаёт записи в ItemProcessor.|
|3|Spring Batch Job|Обогащение данных (Enrichment)|ProductItemProcessor для каждой записи запрашивает в таблице loyality_data актуальные данные по программе лояльности (по ключу productSku) и обновляет поле productData.|
|4|Spring Batch Job|Запись обновлённых данных в БД|JdbcBatchItemWriter сохраняет обработанные записи в таблицу products базы данных номенклатуры PostgreSQL.|
|5|Система|Логирование и уведомление о завершении|JobCompletionNotificationListener фиксирует результат выполнения Job, выводит в лог итоговые данные из таблицы products.|
|6|Java-приложение|Запуск ETL-пайплайна|Бэкенд-приложение запускает Spring Batch Job при старте приложения (в рамках POC) или по триггеру.|

### **Нефункциональные требования**

|**№**|**Требование**|
| :-: | :- |
|1|Пропускная способность: обработка до 400 000 строк в сутки в штатном режиме; в пиковые дни — до 1 200 000 строк.|
|2|Производительность: среднее время обработки файла на 2 000 строк не превышает 30 секунд.|
|3|Параллелизм: поддержка 100–150 одновременных загрузок в пиковые часы (достигается горизонтальным масштабированием через несколько экземпляров Job).|
|4|Надёжность: автоматический повтор (retry) шагов при временных сбоях; возможность перезапуска Job с точки отказа (restart from checkpoint).|
|5|Интеграция: решение должно органично встраиваться в Java/Spring-стек компании и деплоиться вместе с существующим бэкенд-приложением.|
|6|Облачное развёртывание: возможность запуска в GCP (Cloud Run, GKE или Compute Engine) без изменения бизнес-логики.|
|7|Наблюдаемость: готовность к интеграции с системами логирования (ELK) и мониторинга (Prometheus/Grafana) через Micrometer/Actuator.|
|8|Сопровождаемость: чёткое разделение Read–Process–Write упрощает тестирование и поддержку каждого шага независимо.|

### **Решение**

Выбрано **Spring Batch** как технологическое решение для ETL-пайплайна.

#### Обоснование

| Критерий | Spring Batch | Apache Airflow | K8s CronJob | Apache Spark |
|----------|-------------|---------------|-------------|-------------|
| Нативная интеграция с Java/Spring | ✅ Да, часть экосистемы Spring | ❌ Python-ориентирован | ⚠️ Нет встроенной логики | ⚠️ JVM, но другая парадигма |
| Chunk-based обработка (память) | ✅ Встроена | ❌ Нет | ❌ Нет | ✅ RDD/partition |
| Retry и Skip из коробки | ✅ Настраивается на уровне Step | ⚠️ Есть, но через Python DAG | ❌ Нет, нужна ручная реализация | ⚠️ Есть, сложнее настраивать |
| Хранение состояния Job (JobRepository) | ✅ Встроено в PostgreSQL | ✅ Встроено | ❌ Нет | ❌ Нет |
| Перезапуск с точки отказа | ✅ Да | ⚠️ Частично | ❌ Нет | ❌ Нет |
| Ресурсоёмкость | ✅ Низкая | ❌ Высокая (отдельный сервер + БД) | ✅ Низкая | ❌ Высокая (кластер) |
| Сложность развёртывания в GCP | ✅ JAR в контейнере / Cloud Run | ⚠️ Cloud Composer (дорого) | ✅ Прост, но ограничен | ⚠️ Dataproc или GKE |
| Масштабирование под нагрузкой | ✅ Partitioned Step, Remote Chunking | ✅ Dag с параллельными задачами | ⚠️ Через количество Pod | ✅ Нативное |
| Удобство мониторинга | ✅ Micrometer/Actuator/Spring Boot Admin | ✅ Встроенный UI | ⚠️ Только k8s-метрики | ⚠️ Spark UI |

**Вывод:** Spring Batch является оптимальным выбором, поскольку:
- Органично встраивается в существующий Spring/Java-стек TradeWare без изменения архитектуры.
- Обеспечивает декларативное определение пайплайна (Reader → Processor → Writer) с минимальным кодом.
- Предоставляет retry, skip, restart из коробки — критично для надёжности при пиковых нагрузках.
- Chunk-based обработка позволяет работать с большими файлами без загрузки всего объёма в память.
- Развёртывание в GCP сводится к запуску Docker-контейнера (Cloud Run, GKE).

#### Архитектура решения (C4 — Container Level)

Архитектура To Be добавляет компонент **Spring Batch** внутри Java-бэкенда на GCP/ComputeEngine:

```
Сотрудник склада
    │ загружает CSV через браузер
    ▼
[Angular Web App] ──REST──▶ [Java App (WildFly/Spring Boot)]
                                    │
                                    │ запускает Job
                                    ▼
                          [Spring Batch Job]
                          ┌────────────────────────────────────┐
                          │  FlatFileItemReader                 │
                          │      ↓                             │
                          │  ProductItemProcessor               │
                          │      ↓                             │
                          │  JdbcBatchItemWriter                │
                          └────────────────────────────────────┘
                               │          │           │
                    читает CSV │          │ читает    │ пишет
                               ▼          ▼           ▼
                          [GCS]   [БД справочных] [БД номенклатуры]
                                    данных           (products)
                                  (loyality_data)
```

Диаграмма C4 уровня Container (To Be) находится в файле `c4-diagram-tobe.drawio`.

#### Описание шагов ETL-пайплайна

1. **Extract (чтение):** `FlatFileItemReader<Product>` читает `product-data.csv` построчно. Маппинг полей: `productId`, `productSku`, `productName`, `productAmount`, `productData`.
2. **Transform (обогащение):** `ProductItemProcessor` выполняет lookup в таблицу `loyality_data` по `productSku`. Если запись найдена — заменяет `productData` значением `loyalityData` из БД.
3. **Load (запись):** `JdbcBatchItemWriter` сохраняет обработанные записи в таблицу `products` через batch INSERT.
4. **Notification:** `JobCompletionNotificationListener` при успешном завершении Job делает SELECT из `products` и логирует все записи.

Шаги выполняются в рамках одного `Step` с `chunk(3)` — каждые 3 прочитанных элемента формируют один batch-insert.

### **Альтернативы**

#### Apache Airflow
Мощная оркестрационная платформа с веб-UI, поддержкой DAG, встроенным мониторингом и богатой экосистемой провайдеров (BigQuery, GCS, PostgreSQL, Kafka, Spark). Однако требует отдельного развёртывания (Python-сервер + метабаза) и написания логики на Python, что не соответствует Java-стеку TradeWare. Подходит, если в дальнейшем понадобится сложная межсервисная оркестрация.

#### K8s CronJob
Лёгкое решение для периодического запуска контейнеров. Хорошо подходит для простых задач выгрузки без сложной логики обработки. Не имеет встроенной поддержки chunk-processing, retry на уровне шага или хранения состояния выполнения. Требует ручной реализации всей ETL-логики внутри приложения.

#### Apache Spark
Оптимален для обработки данных объёмом от десятков миллионов записей и сложных аналитических трансформаций. Для текущего объёма TradeWare (400K строк/день) является избыточным: требует кластер (Dataproc или GKE с Spark Operator), высокие операционные затраты и кривая обучения.

**Недостатки, ограничения, риски**

| # | Описание |
|---|----------|
|1|**Монолитный запуск:** В текущем POC Job запускается при старте Spring Boot-приложения. В production необходим явный триггер (REST endpoint, Scheduler, или event от GCS). Риск: повторный запуск Job при рестарте контейнера.|
|2|**JobRepository:** Spring Batch хранит метаданные выполнения Job в БД. Требует отдельного управления схемой (таблицы BATCH_*) и периодической очистки истории.|
|3|**SQL Injection в ProductItemProcessor:** Текущая реализация использует конкатенацию строк при формировании SQL-запроса к loyality_data. Необходимо заменить на PreparedStatement или NamedParameterJdbcTemplate.|
|4|**Масштабирование:** При нагрузке 100–150 параллельных загрузок потребуется Partitioned Step или Remote Chunking с разделением файлов. Текущий POC не реализует параллельную обработку.|
|5|**Отсутствие email-уведомлений:** В POC реализовано только логирование. Для production необходима интеграция с SMTP или сервисом уведомлений через `JobExecutionListener`.|
