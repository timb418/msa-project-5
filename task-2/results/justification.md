# Обоснование выбора технологического решения для экспорта прайс-листов

## Контекст задачи

Онлайн-магазин ежедневно в 6:00 генерирует кастомные CSV прайс-листы для B2B-клиентов. Задача: связать данные из четырёх таблиц PostgreSQL (products, categories, clients, client_prices — итого ~20 000 строк) и выгрузить в CSV. Дополнительная обработка данных не требуется. Инфраструктура — микросервисы в Kubernetes (облако).

---

## Сравнительный анализ решений

| Критерий | Spring Batch | Apache Airflow | **K8s CronJob** | Apache Spark |
|---|---|---|---|---|
| **Наличие конфигурации CRON-расписания** | ❌ Нет встроенного CRON — требует внешнего планировщика (Quartz, `@Scheduled`, K8s CronJob) | ✅ Нативный CRON через `schedule_interval`, управление через Web UI | ✅ Нативный CRON в `spec.schedule`, управление через `kubectl` | ❌ Нет встроенного CRON — требует внешнего оркестратора |
| **Сложность реализации логики обработки данных** | Высокая гибкость — Reader/Processor/Writer, чанкинг, skip/retry на уровне записей; для простого JOIN избыточна | Высокая гибкость — Python-операторы, ветвление, XCom; для простого JOIN избыточна | Минимальная — произвольный скрипт или приложение в контейнере; идеально для JOIN + CSV | Высокая — оправдана при сложных трансформациях миллионов строк; для 20K избыточна |
| **Ресурсоёмкость решения** | Средняя — JVM overhead, Spring Boot контейнер постоянно запущен или стартует при каждом Job | Высокая — Webserver + Scheduler + Worker + метабаза PostgreSQL (минимум 3–4 контейнера) | Минимальная — Pod существует только во время выполнения, ресурсы потребляются ~1–2 мин в сутки | Высокая — Driver + Executors, требует отдельного кластера или Spark Operator |
| **Масштабируемость под нагрузкой и сложность реализации** | Горизонтальное партиционирование через Remote Partitioning — сложно настроить; для 20K строк не нужно | CeleryExecutor + Redis — горизонтальное масштабирование воркеров; для одного джоба избыточно | `parallelism` и `completions` в Job spec — нативная горизонтальная параллельность; для 20K строк достаточно одного Pod | Нативная горизонтальная обработка больших данных; для 20K строк не нужно |
| **Сложность развёртывания в облаке и интеграция с MSA** | Средняя — Docker-образ → K8s Deployment, хорошо интегрируется, но требует отдельного сервиса | Высокая — Cloud Composer (GCP) / MWAA (AWS) — отдельная управляемая инфраструктура, дополнительные расходы | Минимальная — нативный объект K8s, никакой новой инфраструктуры, единый namespace с MSA, те же Secret/ConfigMap | Высокая — Dataproc/EMR требует отдельного кластера или Spark Operator в K8s |
| **Удобство интеграции с логированием и мониторингом** | Хорошая — Spring Actuator + Micrometer → Prometheus/ELK, требует конфигурации | Хорошая — встроенный Web UI, Prometheus-экспортёр, логи в метабазе | Хорошая — логи в stdout → ELK (Filebeat), метрики через kube-state-metrics, kube_job_status_* метрики нативны | Средняя — Spark History Server, JMX, требует отдельной конфигурации |

## Вывод: Kubernetes CronJob

**Kubernetes CronJob** — оптимальный выбор для данного кейса по следующим причинам:

### 1. Нулевые накладные расходы на инфраструктуру

Инфраструктура уже в Kubernetes. CronJob — нативный объект K8s (`batch/v1`), не требует развёртывания никаких дополнительных сервисов. Достаточно добавить один YAML-манифест в существующий кластер.

### 2. Минимальное потребление ресурсов

Pod существует только во время выполнения (~1–2 минуты в сутки). В остальное время ресурсы не потребляются. При объёме 20K строк с простым JOIN это особенно важно — платить за постоянно работающий оркестратор (Airflow) или ETL-сервис (Spring Batch) экономически нецелесообразно.

### 3. Полное соответствие требованиям

Задача — JOIN таблиц + запись CSV, никакой обработки данных. K8s CronJob запускает любой контейнер по расписанию. Встроенная retry-политика (`backoffLimit`), таймаут (`activeDeadlineSeconds`), хранение истории запусков (`successfulJobsHistoryLimit`, `failedJobsHistoryLimit`) — всё из коробки.

### 4. Единая операционная модель с MSA

Те же инструменты управления (`kubectl`, Helm, GitOps через ArgoCD/Flux), те же Secret и ConfigMap для конфигурации, та же сетевая политика. DevOps-команда не изучает новые инструменты.

### 5. Нативный мониторинг

kube-state-metrics, уже присутствующий в большинстве K8s-кластеров, экспортирует метрики `kube_job_status_succeeded` и `kube_job_status_failed` без дополнительной настройки. Grafana-дашборд и алерт на упавший Job настраиваются за 15 минут.

## Описание компонентов

| Компонент | Тип | Описание |
|-----------|-----|----------|
| **K8s CronJob** | Kubernetes object | Запускает Export Job Pod каждое утро в 06:00 по расписанию `0 6 * * *`. Управляет retry (`backoffLimit: 3`) и таймаутом (`activeDeadlineSeconds: 1800`) |
| **Export Job Pod** | Kubernetes Pod | Контейнер с экспортным приложением. Подключается к PostgreSQL, выполняет JOIN-запрос, формирует CSV-файл для каждого клиента и загружает в Object Storage |
| **PostgreSQL** | Container (existing) | Источник данных. Таблицы: `products`, `categories`, `clients`, `client_prices` |
| **Object Storage (GCS/S3)** | External System | Хранилище сгенерированных CSV-файлов. Файлы именуются по схеме `price_list_{client_id}_{date}.csv` |
| **Secret** | Kubernetes object | Учётные данные БД (`DB_PASSWORD`) и ключи доступа к Storage (`STORAGE_KEY`) в base64 |
| **ConfigMap** | Kubernetes object | Параметры конфигурации: `DB_URL`, `STORAGE_BUCKET`, `OUTPUT_PATH` |
| **Filebeat DaemonSet** | Kubernetes DaemonSet | Собирает stdout-логи контейнеров и отправляет в Elasticsearch |
| **Prometheus + kube-state-metrics** | Monitoring | Отслеживает метрики Job: `kube_job_status_succeeded`, `kube_job_status_failed`, `kube_job_completion_time` |
| **Grafana** | Monitoring UI | Дашборд истории запусков CronJob, алерт при `kube_job_status_failed > 0` |

## Альтернативы и почему они не выбраны

- **Spring Batch** — хорош для сложного ETL в Java-экосистеме с чанкингом, retry на уровне записей и многошаговыми пайплайнами. Для данной задачи избыточен: нет обработки данных, достаточно одного SQL-запроса. Используется в Tasks 4–6 именно там, где нужен полноценный ETL. Становится предпочтительным, если в будущем потребуется усложнение логики или уведомления по email.

- **Apache Airflow** — идеальный оркестратор для сложных DAG с ветвлением, event-triggers и интеграциями с BigQuery/Kafka (выбран в Task 1 именно по этим причинам). Для одного ежедневного джоба без зависимостей разворачивать Webserver + Scheduler + Worker + метабазу — неоправданно дорого в эксплуатации.

- **Apache Spark** — оправдан при обработке десятков миллионов строк с комплексными трансформациями. Объём 20K строк Spark обработает за те же секунды, что и простой скрипт, при многократно более высоком overhead на запуск кластера.
